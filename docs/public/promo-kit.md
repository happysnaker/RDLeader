## Core angle

RDLeader is a local-first control plane for AI R&D workers. It focuses on the operational layer around agents: task ownership, context routing, runtime dispatch, result collection, approvals, and QA evidence.

## One-line descriptions

- RDLeader is a local-first manager console for supervising multiple AI R&D workers with runtime evidence and approval boundaries.
- Not another chat UI: RDLeader is a small control plane for task ownership, runtime results, and human-in-the-loop agent operations.
- RDLeader explores what happens after “run an agent”: who owns the task, what context was loaded, what result came back, and what needs approval.

## LinkedIn draft

I’m working on RDLeader as a local-first control plane for AI R&D workers.

The problem I care about is not “can an agent answer a prompt?” It is what happens after you have several workers running:

- Who owns which task?
- What context was loaded?
- Did the runtime actually produce a result?
- What is blocked?
- Which actions require human approval?
- What evidence proves the system is still healthy?

The public repo now has a DevPlan-safe README, a public release roadmap, sanitized QA evidence, and a runtime endurance model. I’m intentionally not publishing raw local logs because they can include private paths, identifiers, and live integration traces.
